The five substantive conflicts, and how this guide resolves each

1
Does the Assessment Hub exist as a distinct surface?
No — deleted. Complete Guide had it as Layer 2; Phase 3 Review recommended retiring it; v8 has acted on that. The functions it used to hold (read card data, compute composites, render Laporan) are now split: reading and rendering happen inside each Laporan HTML; composite calculation happens inside the Card before save.
v8 wins
2
Where does the Laporan Psikologis live?
Now: standalone HTML file per battery — one file per A/B/C/D — with Manager View on top and Psikolog View expandable below. Phase 3 target: two React pages inside the Myralix portal (BatterySessions.jsx list + BatteryResultDetail.jsx), with the HTML files retained only as offline fallback until the portal pages are stable, then retired.
v8 now
P3 target
3
Where does score normalisation happen (1–10 composite → 0–100 three-pillar)?
Now: each Laporan's calc3Pillar() does this on read. Phase 3 target: backend normaliseScores(), called once on submit, result stored in battery_results.scores_normalised. The output format (Cognitive / Personality / Work Attitude, each 0–100, plus Overall) is unchanged.
v8 now
P3 target
4
Does the battery architecture own Medical Check (MCU)?
No. MCU is a parallel track inside the Myralix ATS platform (Layer 3), not part of the battery system. The Platform runs the dual gate (Psych AND MCU must pass); the Card and Laporan are unaware of MCU. This matches Complete Guide's framing and is not contradicted by v8 or Phase 3 Review — they simply didn't cover it.
Complete
5
What is the role of the Biznet NEO server?
Today: idle as far as the battery system is concerned (everything is client-side). Near-term (Phase 2b): serve Battery HTMLs at stable URLs via Nginx. Phase 3: run the full backend — Express API, PostgreSQL, Anthropic proxy — eliminating the exposed API key and the localStorage bridge simultaneously.

03
Data contracts & storage v8
Two localStorage keys today — one REST schema in Phase 3

The three components exchange data through exactly two keys. The Card is the sole writer of scores; the Laporan owns the report state; the Platform only reads. This single-writer rule is what keeps the model honest — and it is what makes the Phase 3 migration to a backend so mechanical: each key becomes a table, each access pattern becomes an endpoint.

📊 Candidate scores

myx-bat-{a|b|c|d}-v8
WRITE
Candidate Card — on test completion, saves profile + raw per-instrument scores + computed composites
READ
Platform — to display 3-pillar scores in the Assessment Results Table
READ
Laporan — to build the report content (3-pillar bridge + per-instrument detail sections)
RULE
Never modified by Platform or Laporan — this key is the source of truth for what the candidate answered.

📝 Recruiter decisions

myx-rpt-{a|b|c|d}-v8
READ + WRITE
Laporan — stores qualitative ratings per instrument, edited AI narratives, final verdict, assessor notes
READ
Platform — reads final verdict to update candidate status in the pipeline
RULE
Candidate Card has no knowledge of this key — ensures candidates cannot see recruiter notes or verdicts under any circumstance.

Phase 3 mapping. myx-bat-{x}-v8 becomes the battery_results table (JSONB raw_results column + typed scores_normalised column). myx-rpt-{x}-v8 becomes the laporan JSONB field on the same row, plus a structured final_verdict column. The single-writer rule translates directly into API write permissions: the public POST /api/battery/submit endpoint writes raw_results; authenticated recruiter endpoints update laporan and final_verdict.
04
The scoring bridge v8Complete
From 1–10 battery composites to 0–100 three-pillar output for Platform display

The Card stores raw instrument scores and battery composites on a 1–10 scale (psychometric convention). The Platform's Assessment Results Table expects a 0–100 scale across three dimensions — Cognitive, Personality, Work Attitude — plus an Overall. The calc3Pillar() function bridges the two. It lives inside each Laporan today and moves to backend/src/modules/battery/service.ts :: normaliseScores() in Phase 3.

Pillar Instruments contributing Scale in Scale out Default pass Treatment
Cognitive WPT × 65% + DAT-CS × 35% (Battery A); broader DAT subtest set for B/C/D 1–10 composite 0–100 ≥ 70 Hard gate. Fully weighted as TK: 1.0 in the overall composite.
Personality Big Five (BFI-44) + DISC (A); + EPPS / 16PF / PAPI per battery Per-dimension % 0–100 aggregate ≥ 65 Profile-only. profileOnly: true, weight: 0. Surfaced qualitatively as ✅ Sesuai / ⚠️ Pertimbangkan / ❌ Tidak Sesuai — not composited.
Work Attitude Holland RIASEC + DISC + (SJT / PAPI Kostick for B+/C+) Mixed 0–100 ≥ 65 Weighted blend — weights per battery in SCORING.json.
Overall Cognitive hard gate + weighted personality/work-attitude signal — 0–100 ✅ Pass / Review / Fail Drives the Platform status chip. Written back to Laporan as finalRec.
🎯
Architectural principle — two-layer scoring. Cognitive is a quantitative gate. Personality and behavioural instruments (GPP-I, DISC, 16PF, EPPS) are qualitative profiles, not composite inputs. This was the confirmed architectural direction in earlier iterations and remains correct: personality instruments measure fit, not ability, so they should not be arithmetically composited into a pass/fail score. The Laporan surfaces them as recruiter judgement prompts.

II
Part Two
Wider platform context
What's being built around the battery system — the full ATS pipeline, MCU dual-track, and combined gate
06
The full ATS pipeline Complete
Assessment is stage 4 of 6 — the battery system is one stage inside a larger product

The battery system is not the whole product. Myralix's six-stage ATS pipeline places Assessment after Interview and before BG Check. The v8 Guide and Phase 3 Review focus tightly on the Assessment stage; the Complete Architecture Guide provides the wider context, which is included here so decisions inside the battery system stay aligned with what happens upstream and downstream.

Full Myralix pipeline. The battery system sits inside stage 4 and handles the psychological half of Assessment — the medical half is handled by external lab partners and aggregated in the Platform.
07
Assessment is a dual-track gate Complete
Psychological (this battery system) + Medical (external labs) — both must pass

Inside the Assessment stage, two parallel tracks run simultaneously to shorten time-to-hire by 3–5 days. Track A is the psychometric battery this architecture describes. Track B is a medical check (MCU) scheduled with external partners — Prodia, Kimia Farma, etc. Both must clear before a candidate advances to BG Check. This dual-track logic lives entirely in the Platform; the Card and Laporan know nothing about it.

The battery system (Track A) is fully described in Part I. The MCU track (Track B) is external; its results flow into the Platform's Medical Check tab, independent of anything the Laporan knows about. The gate is enforced by the Platform, and only opens when both tracks return a pass.

Why this matters for the battery architecture. It explains why the Laporan only outputs a psychological verdict (Direkomendasikan / Pertimbangkan / Tidak) rather than a pipeline advancement action — advancement requires MCU clearance as well, and that's a decision only the Platform can make. It also explains why the battery system is deliberately scoped narrow: trying to put MCU awareness into the Laporan was an earlier temptation and is explicitly ruled out.

11
Four gaps the Phase 3 plan doesn't call out Phase 3
Production hardening the backend module needs before launch

The Phase 3 Implementation Guide's backend is solid on happy-path shape but silent on four production concerns. These are small changes individually, but skipping any one of them means the module is not actually production-ready.

Gap What it is Why it matters & what to add
1. Rate limiting POST /api/battery/submit is public on the open internet with no throttle. A trivial DDoS target. Add express-rate-limit keyed on token (one submit per token anyway) with an IP-based secondary. Block abusive IPs at the Apache layer as the second line of defence.
2. Battery-type validation The submit controller accepts whatever payload the Card sends without checking it matches the session's battery. Someone could submit a Battery B payload against a Battery A token. Add a single check in submitResults(): if (payload.battery !== session.battery) throw 400. Logs the mismatch for monitoring.
3. B/C/D score normalisation normaliseScores() is written for Battery A only. B, C, D silently fall through to zeros. Silently wrong is worse than loudly missing. Either implement all four normalisations before launch, or throw on unknown battery types. Never ship with the silent-zero failure mode intact.
4. RBAC registration Routes are mounted behind authMiddleware but the "Battery" module is not registered in the RBAC seed. Any authenticated user can hit every endpoint. Add Battery to the permissions module list with per-role grants (recruiter: read + create invite; psikolog: read + edit Laporan; admin: all).

Laporan evolution timeline Integrated
The single most contested architectural decision across the three documents — resolved here

The Complete Guide treated the Laporan as embedded inside an Assessment Hub. The v8 Guide has them as first-class standalone HTMLs, one per battery. The Phase 3 Review argued they should become React pages in the portal. All three are defensible from their vantage point; the honest path is a staged evolution that starts where v8 is today and ends where Phase 3 Review wants to be.

Phase 2 (current)
Standalone HTML per battery
Four Laporan HTMLs (Myralix_Battery_{A,B,C,D}_Report.html). Manager View and Psikolog View merged into one file. Reads both localStorage keys; writes myx-rpt-{x}-v8. Opened from Platform's 📄 icon. Anthropic calls direct from browser (key exposed — fix per Section 10).
Phase 2b (near-term)
Same HTMLs, served from Biznet
The same four files, now hosted at stable URLs via Nginx on the Biznet instance (e.g. assess.myralix.id/report/A). Eliminates the file-emailing workflow. Still client-side; still localStorage-backed. A one-week, low-risk infrastructure change that unlocks candidate access from anywhere.
Phase 3 (target)
React portal pages + HTML retired
Two React pages inside the Myralix portal become the canonical surface: BatterySessions.jsx (list) and BatteryResultDetail.jsx (report + review). Data comes from PostgreSQL via JWT-authenticated API. Printable via browser or server-side PDF export. HTMLs retained only as offline fallback during the transition, then deleted once the portal detail page prints cleanly.
🎯
Why not skip Phase 2b? Because it's a 1-week change with an immediate UX payoff (no more file email), zero dependency on Phase 3 work, and it creates the assess.myralix.id vanity domain that Phase 3 will reuse for the candidate token URLs. Doing it in sequence means the domain is already live when Phase 3 launches.

The one-page summary Integrated
If you read nothing else in this document, read this
📌
The architecture in one paragraph. Myralix's psychometric assessment is delivered through three components: the Platform (recruiter dashboard, orchestrates the pipeline), the Candidate Card (4 HTMLs, one per battery, administers tests with obfuscated labels), and the Laporan Psikologis (4 HTMLs, one per battery, translates scores into a narrative verdict). They exchange data through two localStorage keys today, and through PostgreSQL + REST in Phase 3. The battery system sits inside stage 4 of a 6-stage ATS pipeline, where it runs in parallel with a medical check (MCU) that the Platform — not the battery — owns. Cognitive is a hard quantitative gate (0–100, ≥70); personality instruments are qualitative profiles, not composited. The migration to Phase 3 is a 4-week plan on a single Rp 725K/month Biznet instance, with two one-hour prerequisite fixes that ship this week regardless.
Where we came from
Hub-centric world (Complete Guide)
Three layers with a standalone Assessment Hub as Layer 2, embedding the Laporan inside it. MCU dual-track and Biznet infrastructure introduced here. The Hub concept has since been retired.
Where we are (v8, Apr 2026)
Three components, two keys
Platform ↔ Card ↔ Laporan. Two localStorage keys. Laporan is a standalone HTML per battery with Manager+Psikolog views merged. API key still in browser HTML — fix this week.
Where we're going (Phase 3)
Biznet-hosted backend + portal pages
Backend module on existing Biznet instance. PostgreSQL replaces localStorage. Two React portal pages replace Laporan HTMLs. Four production gaps closed. API key lives only in .env.
