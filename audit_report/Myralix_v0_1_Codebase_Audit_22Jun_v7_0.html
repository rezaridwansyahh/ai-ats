§0 — Audit methodology & transparency notes
What this audit can and cannot confirm

This section is new in v7.0. It exists because previous audits were not fully transparent about the method. Please read it before relying on any figure in this document.

🔍 How this audit works — what "GitHub connection" actually means

The codebase is not read via a live GitHub API. It is read via project knowledge — a snapshot of your repository that was indexed when your team added it to this Claude project. That snapshot is searched using semantic queries each time an audit is run.

What this means for you:

  • If files were added to the project knowledge recently, they will appear. If commits were pushed after the last sync, I cannot see them.
  • I cannot tell you the exact date the snapshot was last updated — I do not have access to that metadata.
  • Every file I cite in this audit is one I have directly read from the indexed content. I do not invent file names or claim files exist without seeing them.
  • If no new files are visible since last week, the audit will correctly show no change — not invent progress.

Recommendation: To get a fully accurate weekly delta, your team should push the latest codebase to the project knowledge before each audit meeting. This takes ~2 minutes and ensures the audit reflects this week's actual commits, not last week's.
